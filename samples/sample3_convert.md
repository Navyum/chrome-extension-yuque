## 背景

**LLM 写代码很快，但一旦面对长远规划或复杂上下文，就容易“幻觉、降智、迷失方向”**。

修 bug 一直是个头疼的问题，没有需求这么体系化，却又需要耗费大量人力资源进行处理。为了构建一个自动化的 Bug 修复 Agent，对于代码生成和修复，我们不能仅仅依赖大语言模型（LLM）的自由发散，而是必须将其嵌入到具有强规则、确定性校验、沙箱执行和严格状态机流转的**硬性工作流**中，从而收敛 LLM 的幻觉，保证修复的可靠性。

Compound AI Systems：**当前 Agent 发展的核心理论范式**。

-   **核心理念：** 解决复杂工程问题，不能靠“一个巨大的模型 + 一段巨大的 Prompt（Omni-Bot 模式）”，而是要构建一个**复合系统**。
-   **系统组件：** 这个系统应该包含：模型（LLMs）、外部工具（编译器、Linter、搜索）、检索器（RAG 查阅代码库）、控制流（状态机、If-Else 规则拦截器）。

## Demo

bug 自动化流程，只针对 bug，比如产运或者研发发现当前项目存在 bug，可以通过飞书的方式提出，agent 将 bug 收集到列表进行统一管理，相当于一个待办清单，然后下游 agent 能够拉取未处理 bug，先对 bug 进行可行性分析，结合项目查看这个 bug 是否为真，如果是误报或者无法通过代码修复等情况就返回，如果可以修复就交给执行 agent 进行拉分支，写代码，写单测，相当于一个研发 agent，然后再交给测试 agent 进行二次处理，包括部署到测试环境联调等内容，联调通过后会给到运维 agent 进行上线，其中会有哨兵 agent 进行整个流程的管控，如果遇到某个 agent 出现错误或者流程终止需要对清单的任务状态进行处理变更和通知，相关于管理全局，其中也需要对整个流程进行监控。

## 状态流转

状态枚举：

| 阶段 | 做什么 | 谁做 |
| --- | --- | --- |
| **NEW** | 刚提交，待处理 | 人工 |
| **ANALYZING** | 搜索代码，判断 Bug 真伪 | agent: `bug-analyze` |
| **FIXING** | 独立分支修复代码 + 伴随测试，迭代到 pytest 通过 | agent: `bug-dev` |
| **TESTING** | 独立代码审查 + 边界测试，双重验证修复质量 | agent: `bug-test` |
| **DEPLOYING** | 把修复分支部署到测试环境 | agent: `bug-deploy` |
| **INTEGRATION** | 跑 E2E / 接口联调测试，验证真实环境行为 | agent: `bug-integration` |
| **REPORTING** | 汇总所有阶段结果（含干预记录），生成修复报告 + 分支信息 | Supervisor: `bug-orchestrator` |
| **DONE** | 终态，修复完成，报告和分支已就绪供人工 check | — |
| **REJECTED** | 终态，判定非 Bug | — |
| **FAILED** | 某阶段失败，attempt < 3 时回退重试，否则保持 FAILED | Supervisor: `bug-orchestrator` |
| **PAUSED** | 人工打断，记录干预原因，等待决策后恢复或改判 | 人工 |

## PAUSED 机制

设计原则：

任意状态都可能被人工打断——包括 DONE 和 REJECTED 等"终态"。人工无法实时盯着整个执行过程，流水线跑完后的事后审查同样可能发现问题。

打断的本质都是同一件事：**暂停流程、记录问题、等人工决策、恢复或改判**。因此采用统一的 `PAUSED` 状态 + `paused_stage` / `error_origin_stage` 字段，而非为每个阶段设置独立打断状态。

PAUSED 的定位：

1.  **暂停机制** — 阻止 Agent 继续执行，等待人工决策
2.  **质量改进数据入口** — 每一次人工干预都沉淀为 intervention 记录，驱动 Skill 持续优化

## 项目

### GSD （Get Shit Done） - 核心约束：【上下文隔离与环境纯洁度】

GSD 最初是一个在 GitHub 上爆火的提示词框架，目前已经进化为基于 TypeScript 的强力 Agent 编排系统（支持 Claude Code， Cursor， Copilot 等十几种运行时）。

-   **它要解决的核心痛点：****Context Rot（上下文腐化）**。当 Agent 在同一个会话里不断修改代码、看报错、再修改时，Token 会越来越长，模型的智商会断崖式下降（通常上下文占用率过半后质量锐减）。
-   **它的实现机制（Hardness）：** \* **彻底的上下文隔离**：它将大任务拆解为原子任务（Atomic Tasks）。每一个代码修改任务，GSD 都会为其**拉起一个全新的、极其干净的 200K 上下文窗口**，只加载与该任务绝对相关的少量文件。

-   **波次执行（Wave-based Execution）**：多个无依赖的任务并行执行，有依赖的排队执行。
-   **强制验收测试**：每个原子任务必须提供类似 `<verify>` 的验证命令（如 `pytest`），通不过就不允许修改全局状态（`STATE.md`）。

### GStack - 核心约束：【角色认知与决策视角】

GStack 是由 Y Combinator （YC） 总裁 Garry Tan 主导开源的框架（在 2026 年极度流行）。它的受众主要是希望把自己“分身”成一整个研发团队的超级个体。

-   **它要解决的核心痛点：** 认知混乱（Cognitive Mush）。如果让同一个模型既做架构决策，又写底层逻辑，它往往会给出平庸或跑题的结果。
-   **实现机制：**

-   **认知齿轮（Cognitive Gears）**：通过极其严格的目录结构（`.claude/skills`）和 `CLAUDE.md` 配置，强制划分角色。
-   它提供了一套斜杠指令流：比如思考用 `/office-hours`，规划用 `/plan`，审核用 `/review`，发布用 `/ship`。
-   当 Agent 处于“Staff Engineer（资深工程师）”模式时，它被加上了硬性约束：不准随便重构、必须遵循某某设计模式。当处于“Review”模式时，它会唤醒多个对抗性子 Agent 分别检查正确性、可维护性和安全性。

### Multica.ai - 核心约束：【企业级状态管理与生命周期】

这更偏向于一个商业化/开源平台化的解决方案，主要关注 Agent 在真实企业环境中的落地。

-   **它要解决的核心痛点：** Agent 很容易中途掉线、死循环，或者无法与企业现有的权限和审批流结合。
-   **实现机制：**

-   **长时自主运行状态机**：支持 Agent 运行数小时不间断，并且所有运行进度（State）全部持久化，网络断开也能恢复。
-   **可靠的治理体系**：划定 Agent 的权限边界（比如只能读 Jira，只能提 PR，绝对不能直接合入主干），以及身份认证。

### **Superpowers （Claude Code 另一大主流框架）**

-   **特色：** 强制的 TDD（测试驱动开发）流水线。它要求 Agent 在写任何业务代码之前，必须先写测试。这对于 Bug 修复场景（复现 Bug -> 写出 Failed Test -> 修复 -> Test Passed）是完美的照搬范本。

### 其他

1.  **SWE-agent （普林斯顿大学开源）**

1.  **特色：** 专门针对修复 GitHub Issue（也就是修 Bug）设计的 Agent。他们设计了一套名为 `Agent-Computer Interface (ACI)` 的专属工具集。比如，他们不让 LLM 随意读文件，而是开发了专门的文件分页阅读器；不让 LLM 随意写文件，而是开发了防错的 `search_and_replace` 函数。

2.  **AutoCodeRover （新加坡国立大学）**

1.  **特色：** 强依赖程序的抽象语法树（AST）。它不仅依靠语义搜索，还结合代码的类、方法结构去精确定位 Bug 发生的位置，这在“诊断”阶段能极大地提升成功率。

3.  **Sago （开源 Agent 编排器）**

1.  **特色：** 这是一个 Agent-Agnostic（模型无关）的 CLI 编排器。它的理念是：用便宜的模型（如 Qwen 或 GPT-4o-mini）去写规划（Plan），然后把具体的执行计划丢给强力模型（Claude 3.5 Sonnet / 3.7）去执行，并且每一步都带有强制的 `<verify>` 校验门。这可以帮你的系统降低 API 成本。
